# Components

This document defines where components live, how they expose their API, and how they compose.

## Three layers

```
┌─────────────────────────────────────────────────────────┐
│ frontend/src/views/                                     │ ← page-level, route targets
├─────────────────────────────────────────────────────────┤
│ frontend/src/design/patterns/                           │ ← domain composites (our code)
├─────────────────────────────────────────────────────────┤
│ frontend/src/design/ui/                                 │ ← shadcn-vue copy-in (CLI generated)
├─────────────────────────────────────────────────────────┤
│ Reka UI primitives + Tailwind tokens (npm dependencies) │
└─────────────────────────────────────────────────────────┘
```

### `design/ui/` — shadcn-vue copy-in

Generated by the shadcn-vue CLI (`npx shadcn-vue@latest add <component>`). They wrap Reka UI primitives with Tailwind classes.

Rules:

- **Do not modify** without a strong reason. We try to keep them close to upstream so future CLI updates apply cleanly.
- If you need a different visual variant, extend via `cva` variants in the same file or wrap in a pattern — do not fork the file.
- Each `add` lands in its own subdirectory with an `index.ts` re-export.
- Every component needs a Vitest spec in `__tests__/` next to the source covering: render with defaults, all variants, slots, disabled, emits.

### `design/patterns/` — our domain composites

Our code, specific to Inventario domain. Composes from `design/ui/` primitives plus app data.

Examples: `LocationCard`, `CommodityCard`, `MediaGallery`, `FileUploader`, `AppHeader`, `EmptyState`, `StatusBadge`.

Rules:

- One pattern per file.
- Props typed via `defineProps<Props>()` (TS form).
- Emits typed via `defineEmits<Emits>()`.
- Slots typed via `defineSlots<Slots>()` when more than the default slot exists.
- No business logic — patterns receive data via props and emit user actions. Data fetching belongs to `services/` + `stores/` consumed by the parent view.
- Each pattern needs a Vitest spec.

### `views/` — page-level

Route targets (under `frontend/src/views/<feature>/`). Compose patterns + ui primitives. **No inline `<style>` blocks**, no inline `<div class="commodity-card">` markup duplicating a pattern.

Rules:

- Imports only from `@design/{ui,patterns}`, composables, services, stores, types.
- No `primevue/*` / `primeicons` / `@fortawesome/*` imports — ESLint blocks them (see [`imports-and-bans.md`](./imports-and-bans.md)).
- Page wiring (route params, data fetching, error handling, loading state) lives here.

## Props

```ts
type Props = {
  location: Location
  totalValue?: { amount: number; currency: string }
  selected?: boolean
}
const props = withDefaults(defineProps<Props>(), {
  selected: false,
})
```

- TypeScript types are the source of truth — no runtime validators.
- Every prop has either a sensible default (via `withDefaults`) or is required.
- Boolean props default to `false`. The presence of `<MyComp selected />` toggles it on.
- Avoid `Object` / `Array` / `any` — always a precise shape.

## Emits

```ts
type Emits = {
  view: [id: string]
  edit: [id: string]
  delete: [id: string]
}
const emit = defineEmits<Emits>()
```

- Emit names use `kebab-case` in templates (`@view`, `@edit`); the TS shape uses `camelCase` keys when needed.
- Payload types are explicit. No `any[]`.

## Slots

```ts
defineSlots<{
  default?: () => unknown
  actions?: () => unknown
}>()
```

- Always define slots when a component has any non-default slot.
- Document the expected slot contract in a JSDoc above `defineSlots` if it is non-trivial.

## Variants via `cva`

Any component with two or more visual variants uses `class-variance-authority`:

```ts
import { cva, type VariantProps } from 'class-variance-authority'
import { cn } from '@design/lib/cn'

const buttonVariants = cva(
  'inline-flex items-center justify-center rounded-md text-sm font-medium transition-colors focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring',
  {
    variants: {
      variant: {
        default: 'bg-primary text-primary-foreground hover:bg-primary/90',
        destructive: 'bg-destructive text-destructive-foreground hover:bg-destructive/90',
        outline: 'border border-input hover:bg-accent',
      },
      size: {
        default: 'h-10 px-4',
        sm: 'h-9 px-3',
        lg: 'h-11 px-8',
      },
    },
    defaultVariants: { variant: 'default', size: 'default' },
  },
)

type Props = VariantProps<typeof buttonVariants> & { class?: string }
const props = defineProps<Props>()
```

Forbidden alternatives:

- `if`/`switch` chains assembling class strings in `<script>`.
- Conditional `:class="{ red: x, blue: y }"` for visual variants (acceptable for dynamic state like `selected`).
- Inline ternaries longer than ~2 conditions.

## Reactivity

- Prefer `computed()` over watchers when deriving values.
- `watch()` for explicit dependency lists; `watchEffect()` only for side-effects with implicit deps.
- `ref()` for primitive state; `reactive()` only for grouped state where you never reassign the wrapper.
- Cleanup: every `onMounted` that registers a listener has a paired `onUnmounted`.

## Composition

Extract logic into `@design/composables/use*` when:

- The same logic appears in two or more components.
- The logic spans multiple lifecycle hooks.
- The logic owns reactive state others can subscribe to.

A composable's contract is defined by its return value; document what each returned ref/function does in JSDoc.

## DOM / data-testid

Every interactive top-level element accepts an optional `data-testid` prop and forwards it to its outermost element. Patterns expose a typed `testId?: string` prop.

Example:

```vue
<script setup lang="ts">
type Props = { testId?: string }
defineProps<Props>()
</script>

<template>
  <Card :data-testid="testId">…</Card>
</template>
```

This keeps Playwright tests stable across DOM refactors (see [`testing.md`](./testing.md)).

## Legacy class anchors

During the design-system migration, certain CSS classes (`.commodity-card`, `.location-card`, `.file-card`, `.file-item`, `.export-row`) act as **marker anchors** for existing Playwright tests. Any new pattern that replaces a legacy component must keep the relevant class on its outermost element:

```vue
<template>
  <Card class="commodity-card group cursor-pointer hover:shadow-md ...">
    …
  </Card>
</template>
```

The class does no styling — Tailwind utilities do. It is only a stable selector for legacy e2e. See [`migration-conventions.md`](./migration-conventions.md) for the full list and removal policy.
